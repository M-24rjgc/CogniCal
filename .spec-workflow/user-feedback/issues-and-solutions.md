### 问题 1：首页仪表盘功能过于冗杂 🟠 P1

**来源**：用户反馈

**问题描述**：

- 用户反馈首页仪表盘"过分冗杂"
- 功能太多导致界面不清晰
- 用户无法快速找到核心功能
- 信息密度过高，视觉疲劳

**当前状态**：
首页 `src/pages/Dashboard.tsx` 包含 4 个主要模块：

1. 工作负载预测横幅（WorkloadForecastBanner）
2. 分析概览（AnalyticsOverview）
3. 每周健康概览（WeeklySummaryPanel）
4. 健康提醒弹窗（WellnessNudgeToast）

**✅ 最终解决方案：渐进式信息展示 + 可自定义仪表盘**

**实施步骤**：

1. **默认简化首页布局**
   - 默认只显示 2-3 个核心模块：
     - ✅ **必须显示**：今日任务概览（新增）
     - ✅ **必须显示**：生产力评分卡片（简化版）
     - ✅ **必须显示**：即将到期任务提醒（新增）
   - 使用"展开/收起"按钮控制详细信息的显示

2. **新增设置页面 - 仪表盘自定义**
   - 在设置中添加"仪表盘显示设置"选项
   - 用户可选择显示/隐藏以下模块：
     - [ ] 工作负载预测横幅
     - [ ] 生产力评分详情
     - [ ] 每周健康概览
     - [ ] 时间分配图表
     - [ ] 效率洞察
     - [ ] 今日专注时长
   - 全选即为"完整模式"，无需额外开关

3. **添加快速入口区域**
   - 在首页顶部添加常用功能快捷入口
   - 包括：创建任务、查看日历、生成智能规划

**技术实现**：

- 在 `settingsStore.ts` 添加 `dashboardModules` 配置
- 修改 `Dashboard.tsx` 根据设置动态渲染模块
- 使用条件渲染和懒加载优化性能

**预期效果**：

- 新用户看到简洁清爽的首页
- 高级用户可自由定制显示内容
- 减少信息过载，提升专注度

### 问题 2：日程软件界面不够简洁 🟠 P1

**来源**：用户反馈

**问题描述**：

- 用户希望界面"简洁一点会更好"
- 主要指功能数量过多，而非视觉风格
- 界面元素较多，影响专注度

**当前状态**：

- 使用 shadcn/ui 组件库（现代化设计）
- Tailwind CSS 样式系统
- 多个数据维度同时展示

**✅ 最终解决方案：通过仪表盘自定义解决 + 视觉微调**

**实施步骤**：

1. **功能简化已通过问题 1 的方案解决**
   - 用户可通过自定义设置控制显示内容
   - 可以通过"更多"菜单访问隐藏功能

2. **视觉风格微调**
   - 保持现有设计风格，进行细节优化：
     - 统一卡片间距为 16px（当前可能不一致）
     - 统一圆角半径为 8px
     - 保持主色调不超过 4 种
   - 深色/浅色主题保持完全一致的布局

3. **优化低频功能访问方式**
   - 将分析功能保持独立页面访问
   - 顶部导航保持简洁（Dashboard、Tasks、Calendar、Settings）

**技术实现**：

- 调整 Tailwind 配置，统一间距和圆角
- 确保深色/浅色主题的 CSS 变量同步

**预期效果**：

- 界面更加整洁统一
- 用户可按需显示功能，减少视觉负担
- 深浅主题体验一致

### 问题 4：功能不清晰，用户不知道这是做什么的 🟠 P1

**来源**：用户反馈

**问题描述**：

- 新用户不知道各功能模块的作用
- 缺少功能说明和引导
- 专业术语不易理解

**当前状态**：

- 无新手引导流程
- 无功能说明 Tooltip
- 无帮助文档入口

**✅ 最终解决方案：添加交互式引导 + 内置帮助文档**

**实施步骤**：

1. **实现首次启动引导流程**
   - 安装并集成 `@driver.js/react` 或 `react-joyride`（推荐 Driver.js）
   - 设计 5-7 步引导流程：
     - 步骤 1：欢迎界面，介绍 CogniCal 是什么
     - 步骤 2：创建第一个任务
     - 步骤 3：介绍 AI 任务解析功能
     - 步骤 4：查看生产力评分
     - 步骤 5：设置 API Key（可选）
     - 步骤 6：介绍如何自定义仪表盘
     - 步骤 7：完成引导，开始使用
2. **为每个功能模块添加帮助按钮**
   - 在每个卡片右上角添加 "?" 图标
   - 点击后显示 Popover 或 Tooltip，解释该功能的作用
   - 示例：
     - 生产力评分：系统根据您的任务完成情况、准时率等多个维度计算的综合评分
     - 工作负载预测：预测未来 7/14/30 天的工作量，帮助您提前规划

3. **内置帮助文档系统**
   - 创建帮助中心页面 `src/pages/Help.tsx`
   - 使用 Markdown 编写帮助文档
   - 分类：
     - 快速入门
     - 任务管理
     - AI 功能
     - 生产力分析
     - 常见问题 FAQ
   - 在顶部导航或设置中添加"帮助"入口

4. **优化功能命名（可选）**
   - 考虑为专业术语添加副标题：
     - "生产力评分"下方小字：基于完成率和准时率
     - "工作负载预测"下方小字：未来任务量预警

**技术实现**：

- 安装 `npm install driver.js`
- 在 `uiStore.ts` 添加 `showOnboarding` 状态
- 创建 `Onboarding.tsx` 组件管理引导流程
- 帮助文档使用 `react-markdown` 渲染

**时间估算**：1 周

**预期效果**：

- 新用户快速上手
- 功能理解清晰
- 降低学习成本

### 问题 6：循环任务功能不完整 🟡 P2

**来源**：技术分析

**问题描述**：

- 数据库中有 `is_recurring` 和 `recurrence_rule` 字段但逻辑未实现
- 用户无法设置重复规则（每周一、每月第一天等）
- 无法自动生成重复任务实例
- 定期会议、固定习惯等场景需要手动创建多个任务

**✅ 最终解决方案：延后到 v2.0，完整实现标准循环任务系统**

**决策依据**：

- 用户强烈需要但实现复杂度高
- 优先解决 P0 和 P1 问题
- 确保实现时采用标准方案（方案 A）

**v2.0 规划（完整实现）**：

1. **支持 iCalendar RRULE 标准（RFC 5545）**
   - 每日重复（每 N 天）
   - 每周重复（指定星期几，可多选）
   - 每月重复（固定日期 or 第 N 个星期几）
   - 每年重复（指定月日）

2. **支持灵活的结束条件**
   - 永不结束
   - 指定结束日期
   - 重复 N 次后结束

3. **智能实例管理**
   - 提前 30 天自动生成未来实例
   - 支持编辑单个实例 vs 编辑整个系列
   - 删除时可选择删除单个或整个系列

4. **UI 设计**
   - 在任务创建/编辑表单添加"重复"选项
   - 清晰的重复规则配置界面
   - 显示下次重复时间预览

**临时替代方案**：

- 用户手动创建重复任务
- 使用标签标记周期性任务（如 `#每周例会`）
- 通过任务模板快速创建

**时间估算**：2-3 周（v2.0 实施）

### 问题 7：数据库性能可能存在隐患 🟡 P2

**来源**：技术分析

**问题描述**：

- `tasks` 表有 32 个字段，可能影响查询性能
- 部分表缺少分页设计（如 `analytics_snapshots`）
- 未发现 FTS5 全文搜索索引（README 提到但 schema 中未见）
- 长期运行后数据量增大可能导致性能下降

**当前状态分析**：

```sql
-- tasks 表字段过多
CREATE TABLE tasks (
    -- 基础字段 8 个
    -- 时间字段 4 个
    -- AI 字段 11 个
    -- 其他字段 9 个
    共 32 个字段
);

-- 缺少 FTS 索引
-- 未见：CREATE VIRTUAL TABLE tasks_fts USING fts5(...)
```

**潜在风险**：

- 任务列表查询变慢（SELECT \* 拉取 32 个字段）
- 搜索功能性能不佳（LIKE 查询而非 FTS）
- 历史数据累积导致分析查询变慢
- 内存占用增加

**解决方案（待讨论）**：

**方案 A - 优化数据库结构**（推荐）

- 拆分 `tasks` 表：
  - `tasks`（核心字段 12 个）
  - `task_ai_insights`（AI 相关字段）
- 添加 FTS5 全文搜索：
  ```sql
  CREATE VIRTUAL TABLE tasks_fts USING fts5(
      title, description, tags,
      content='tasks', content_rowid='id'
  );
  ```
- 实现数据归档：
  - 保留最近 90 天的 `analytics_snapshots`
  - 旧数据移到 `analytics_snapshots_archive`
- 优化查询：
  - 避免 `SELECT *`，明确指定字段
  - 添加必要的复合索引
- 时间估算：1-2 周

**✅ 最终解决方案：立即优化数据库结构（方案 A）**

**决策依据**：

- 工作量不大（1-2 周）
- 预防性优化，避免未来问题
- 个人使用场景下预计任务量 ≤ 1000

**实施步骤**：

1. **拆分 `tasks` 表，减少字段数量**

   ```sql
   -- 核心 tasks 表（保留 12 个常用字段）
   CREATE TABLE tasks_new (
       id, title, description, status, priority,
       planned_start_at, start_at, due_at, completed_at,
       estimated_minutes, estimated_hours, tags
   );

   -- AI 相关数据独立表
   CREATE TABLE task_ai_insights (
       task_id PRIMARY KEY,
       ai_summary, ai_confidence, ai_complexity_score,
       ai_suggested_start_at, ai_focus_mode,
       ai_efficiency_prediction, ai_cot_steps, ai_cot_summary,
       ai_metadata, ai_source, ai_generated_at,
       FOREIGN KEY (task_id) REFERENCES tasks(id) ON DELETE CASCADE
   );
   ```

2. **添加 FTS5 全文搜索索引**

   ```sql
   CREATE VIRTUAL TABLE tasks_fts USING fts5(
       title, description, tags,
       content='tasks',
       content_rowid='id'
   );

   -- 自动同步触发器
   CREATE TRIGGER tasks_ai AFTER INSERT ON tasks BEGIN
       INSERT INTO tasks_fts(rowid, title, description, tags)
       VALUES (new.id, new.title, new.description, new.tags);
   END;
   ```

3. **实现数据归档机制**
   - 保留最近 90 天的 `analytics_snapshots`
   - 创建归档表 `analytics_snapshots_archive`
   - 定期任务自动归档旧数据

4. **优化查询语句**
   - 修改所有 `SELECT *` 为明确字段列表
   - 添加必要的复合索引
   - 使用 `EXPLAIN QUERY PLAN` 分析慢查询

5. **数据迁移脚本**
   - 编写 Rust 迁移代码
   - 确保数据完整性
   - 提供回滚机制

**技术实现**：

- 在 `src-tauri/src/db/migrations.rs` 添加迁移脚本
- 修改 `task_repository.rs` 适配新表结构
- 更新所有查询语句

**时间估算**：1-2 周

**预期效果**：

- 查询性能提升 30-50%
- 搜索速度显著加快（FTS5）
- 数据库文件大小减小
- 长期运行稳定性提高

### 问题 8：缺少数据导入功能 🟢 P3

**来源**：技术分析 + 用户潜在需求

**问题描述**：

- 当前只有数据导出功能（JSON/CSV）
- 用户无法从其他工具迁移数据到 CogniCal
- 无法批量导入任务
- 切换设备时需要手动重建数据

**影响场景**：

- 从 Todoist、Things、Notion 等工具迁移
- 批量创建任务（从 Excel 导入）
- 多设备同步（当前只能导出再导入）
- 数据恢复（误删除后恢复）

**解决方案（待讨论）**：

**方案 A - 完整的导入导出系统**

- 支持导入格式：
  - CogniCal 原生格式（JSON）
  - CSV（需要映射字段）
  - Todoist 格式
  - iCalendar (.ics) 格式
- 导入时数据验证和冲突处理
- 导入预览：显示将要导入的数据

### 问题 10：缺少性能和错误监控 � P3（优先级降低）

**来源**：技术分析

**问题描述**：

- 当前没有错误追踪系统
- 用户遇到错误时无法自动上报
- 缺少性能监控，无法发现性能瓶颈
- 内测期间问题定位困难

**当前状态**：

- 错误只在开发者控制台显示
- 用户遇到问题需要手动截图反馈
- 无法统计错误发生频率和影响范围
- 无法追踪性能指标（响应时间、内存占用）

**解决方案（待讨论）**：

**方案 A - 集成专业监控服务**

- 前端错误追踪：
  - 集成 Sentry（开源，有免费额度）
  - 自动捕获 JavaScript 错误
  - 记录用户操作路径（面包屑）
  - 捕获网络请求失败
- 后端错误追踪：
  - Rust 集成 Sentry SDK
  - 捕获 panic 和错误日志
- 性能监控：
  - 记录页面加载时间
  - 追踪 Tauri Command 响应时间
  - 监控内存使用
- 隐私保护：
  - 用户可选择是否发送错误报告
  - 敏感信息脱敏
  - 符合隐私政策
- 时间估算：3-5 天

**方案 B - 简单的本地日志系统**

- 不集成第三方服务
- 实现本地日志记录：
  - 前端：console.error 写入本地文件
  - 后端：使用 tracing 库记录日志
- 日志文件保存在应用数据目录
- 用户遇到问题时手动提交日志文件
- 添加"导出日志"功能
- 时间估算：2-3 天

**✅ 最终解决方案：延后到 v3.0（正式版）实施（方案 C）**

**决策依据**：

- 内测用户较少，手动收集反馈可行
- 虽有遇到定位困难的问题，但不影响内测进展
- 优先专注核心功能开发
- 正式版前再实施监控系统

**临时替代方案**：

- 内测期间用户手动截图反馈
- 开发者工具查看控制台错误
- 收集常见问题清单

**v3.0 规划（正式版）**：

- 集成 Sentry 错误追踪（前后端）
- 性能监控（页面加载、API 响应时间）
- 用户可选择是否发送错误报告
- 敏感信息自动脱敏

**当前行动**：

- 完善错误提示信息，方便用户反馈
- 添加版本号和构建信息到 UI
- 保持代码质量，减少错误发生
